# Blog Rewrite — 文章優化改寫

## 工作流程

### Phase 1：分析現況

1. 讀取原文
2. 執行 `smart-blog-analyze` 的分析邏輯，取得目前分數
3. 識別最嚴重的問題（優先處理 🔴 致命 → 🟡 高 → 🟠 中）
4. 呈現分析結果給使用者，確認改寫方向

### Phase 2：保留清單

識別並保留原文中的優點：
- 作者獨特的語氣和風格
- 已經寫得好的段落
- 原創數據和個人經驗
- 現有的內部/外部連結
- 品牌相關的特定用語

### Phase 3：研究補充

如果原文有以下問題，生成 `blog-researcher` agent 補充：
- 統計數據過時（>12 個月）
- 統計數據不足（<8 個）
- 缺少圖片
- 缺少圖表數據

**反幻覺規則同 write 流程。**

### Phase 4：改寫

讀取 `blog/references/content-rules.md`，針對問題清單逐項修改：

#### 結構修正
- 修正標題層級跳級
- 加入缺少的 TL;DR 區塊
- 加入缺少的 FAQ 段落
- 調整過長的段落

#### 內容優化
- 將 H2 開頭改為 Answer-First 格式
- 補充新的統計數據（來自研究報告）
- 加入原創性標記
- 加入內部連結 placeholder

#### 反 AI 偵測修正
- 替換 AI 觸發詞
- 增加句長變化
- 加入修辭問句
- 加入口語化表達

#### Frontmatter 更新
- 更新 `lastUpdated` 為今天日期
- 如缺少 `description`，補充 150-160 字元
- 如缺少 `coverImage`，標記 `[待補充：封面圖]`

### Phase 5：交付

```markdown
## 改寫完成：[標題]

### 分數變化
| 類別 | 改寫前 | 改寫後 | 變化 |
|------|--------|--------|------|
| 內容品質 | [N]/30 | [N]/30 | +[N] |
| SEO 優化 | [N]/25 | [N]/25 | +[N] |
| E-E-A-T | [N]/15 | [N]/15 | +[N] |
| 技術元素 | [N]/15 | [N]/15 | +[N] |
| AI 引用 | [N]/15 | [N]/15 | +[N] |
| **總分** | **[N]/100** | **[N]/100** | **+[N]** |

### 主要變更
- [列出主要修改項目]

### 新增數據
- [N] 筆新統計（✅ [V] / ⚠️ [S]）

### 需要使用者處理
- [ ] 確認語氣是否保持一致
- [ ] 確認 [S] 標記的數據
- [ ] 替換 [內部連結] placeholder
```

---

## 更新模式

當使用者說 `/blog rewrite <file> --update` 或 `/blog update <file>`：
- 專注在新鮮度更新（數據更新、日期更新）
- 保留 70%+ 的原文
- 只替換過時的統計數據
- 更新 frontmatter 的 `lastUpdated`
